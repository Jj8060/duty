## 1. 项目概述与背景

### 1.1 项目简介

值日排班管理系统是一个基于Web的在线值日排班和考核管理平台，旨在帮助组织或团队有效地管理值日排班安排、记录值日考勤情况、进行考核评价和统计分析。

### 1.2 项目目标

- 提供直观的值日排班日历视图，方便查看每周值日组安排
- 支持管理员对每日值日考勤情况进行记录和评价
- 建立完善的评分和惩罚机制，规范值日管理
- 通过连接线上数据库，便于记录并通知成员值日表现
- 实现灵活的管理员权限管理，确保系统安全

### 1.3 适用场景

本系统适用于需要定期进行值日排班的组织或团队，如：
- 学校班级值日管理
- 公司办公室清洁值日
- 实验室值日安排
- 其他需要周期性值日排班的场景

### 1.4 核心特点

- **简单易用**：直观的日历视图和仅支持管理员操作的列表视图，操作与观看根据权限划分
- **权限清晰**：终端管理员、普通管理员、普通用户三级权限
- **功能完善**：涵盖排班、考勤、评价、统计全流程
- **数据透明**：所有用户可查看排班和统计信息
- **灵活管理**：支持值日组调整、额外值日人员添加等功能

## 2. 系统需求分析

### 2.1 功能需求

#### 2.1.1 值日排班查看需求

**需求描述**：
- 系统需要显示每周值日组安排，用户可以查看任意周的值日组
- 日历视图支持按周查看，显示周一到周日的值日情况
- 列表视图下管理员可选择是否包含周末显示
- 日历视图默认所有日期显示
- 支持按学期或学年查看，学期起始日期为2月28日或8月30日

**技术要求**：
- 日期范围支持：2026年3月1日至2030年12月31日
- 支持年份和月份切换
- 支持日历视图和列表视图两种显示模式

#### 2.1.2 值日考勤评价需求

**需求描述**：
- 管理员需要对每天的值日人员进行考勤评价
- 评价内容包括：出勤状态、评分、惩罚天数
- 支持标记全体缺勤和重大活动日

**评价维度**：
- 出勤状态：已到、缺席、不合格、待定、优秀
- 评分：0-4分（0分表示未评分，默认评分为0分）
- 惩罚天数：根据评分和缺勤情况自动计算，也允许管理员手动设置

#### 2.1.3 权限管理需求

**需求描述**：
- 系统需要区分三种用户角色：终端管理员、普通管理员、普通用户
- 终端管理员（账号：ZRWY，密码：goodday）拥有最高权限
- 普通管理员由终端管理员创建和管理
- 普通用户无需划分，默认无登录状态下是普通用户
- 网页默认无登录
- 普通用户只能查看，不能进行任何操作

**权限划分**：
- **普通用户**：查看排班表、查看统计信息
- **普通管理员**：拥有普通用户权限 + 考勤评价、修改密码、查看列表视图
- **终端管理员**：拥有普通管理员权限 + 管理员管理、操作日志查看、删除记录

#### 2.1.4 惩罚机制需求

**需求描述**：
- 系统需要根据出勤情况和评分自动计算惩罚天数
- 支持手动调整惩罚天数
- 支持补值减免惩罚机制

**惩罚规则**：
- 缺勤惩罚：第一次缺勤1天，后续缺勤登记天数额外加2天
- 不合格惩罚：1天
- 评价机制：出勤，代改进，不合格，优秀（优秀对应4分，出勤对应3分，待改进2分，不合格与缺勤1分，0分为未评价，不计入统计）
- 周五缺勤：额外+1天惩罚
- 代值自动惩罚：代值人员获得不合格评价惩罚登记在代值人员而不是原始值日生

#### 2.1.5 统计分析需求

**需求描述**：
- 系统需要提供详细的统计数据
- 支持按组别筛选统计
- 显示每个成员的平均分、惩罚天数、考勤情况

**统计指标**：
- 平均分：计算所有有效评分记录的平均值（即非0分）
- 惩罚天数：累计所有惩罚天数
- 考勤统计：已到、缺席、不合格次数

### 2.2 非功能需求

#### 2.2.1 性能需求

- 页面加载时间：首次加载时间不超过3秒
- 操作响应时间：用户操作响应时间不超过2秒
- 支持的数据量：至少支持1000条考勤记录

#### 2.2.2 可用性需求

- 界面简洁清晰，易于操作
- 支持主流浏览器（Chrome、Firefox、Safari、Edge）
- 响应式设计，支持不同屏幕尺寸

#### 2.2.3 安全性需求

- 管理员登录采用密码验证
- 操作权限严格控制，防止未授权操作
- 终端管理员账号不可被删除

#### 2.2.4 数据持久化需求

- 使用supabase存储数据
- 数据不会因页面刷新而丢失

## 3. 系统功能详细设计

### 3.1 值日组管理

#### 3.1.1 值日组结构

系统预设8个值日组，每组3名成员，共24名成员。

**值日组数据结构**：
```typescript
interface Group {
  id: string;           // 组别ID，格式：group-1 到 group-8
  name: string;         // 组别名称，格式：小组1 到 小组8
  members: Member[];    // 组成员列表
}

interface Member {
  id: string;           // 成员ID，格式：member-{组号}-{序号}
  name: string;         // 成员姓名
  groupId: string;      // 所属组别ID
}
```

#### 3.1.2 值日组轮换规则

- 值日安排按周循环，每周从周一开始
- 默认按照小组1到小组8的顺序循环
- 管理员可以手动调整任意周的值日组
- 周末（周六、周日）默认使用当周的值日组

#### 3.1.3 值日组显示

- 在日历视图和列表视图中显示当前值日组
- 显示组内所有成员的姓名
- 管理员可以在列表视图中点击选择框修改值日组

### 3.2 日历视图功能

#### 3.2.1 月历视图

**功能描述**：
- 以月为单位显示值日排班情况
- 每个日期格子显示该天的值日组和成员
- 显示每个成员的考勤状态（用颜色标识）

**显示内容**：
- 日期格子：显示日期数字、值日组名称
- 成员状态圆点：每个成员对应一个彩色圆点（鼠标悬停显示成员姓名和状态）：
  - 深绿圆点：优秀
  - 浅绿圆点：已到
  - 蓝色圆点：待改进
  - 黄色圆点：不合格
  - 红色圆点：缺席
  - 灰色圆点：待定（未评价）
- 全缺/活动标签：若当天有全体缺勤或重大活动标记，额外显示红色/橙色标签
- 额外值日人员：若有额外值日人员则显示紫色 `+N` 标签，数字代表人数

**操作功能**：
- 点击日期格子打开右下角浮动面板，显示：
  - 当日值日组名称与每个成员的状态、评分、惩罚天数
  - 代值人/还值对象信息（若有）
  - 额外值日人员列表（紫色区块，含原因说明）
  - 管理员快速操作按钮（全体缺勤 / 重大活动 / 重置当天）
  - 每个成员的"评价"/"修改"按钮（仅管理员可操作）
- 支持月份和年份切换
- 支持"回到今天"快速定位

#### 3.2.2 日期范围限制

- 起始日期：2026年2月28日
- 结束日期：2030年12月31日
- 超出范围的日期不显示

### 3.3 列表视图功能

#### 3.3.1 周视图

**功能描述**：
- 以周为单位显示值日排班情况
- 显示一周内每天的值日成员和考勤情况（周一是第一天）
- 支持选择查看不同周的数据

**显示内容**：
- 周选择器：选择年份和周数
- 周末开关：可选择是否显示周末（周六、周日），默认不显示周末
- 日期卡片：显示每天的值日情况
- 成员卡片：显示每个成员的状态和操作按钮

**操作功能**：
- 周选择器：可选择2026-2030年的任意周
- 值日组修改：管理员可以修改当周的值日组
- 添加值日人员：管理员可以添加额外值日人员
- 更改代值人员：管理员可以将当天特定值日生换为其他人，评分记至对应代值人员身上
- 快速操作按钮：
  - 全体缺勤：标记当天全体成员缺勤
  - 重大活动：标记当天为重大活动日（值日失效，不计入总评）
  - 重置当天：清除当天的所有评价记录
  - 添加人员：添加额外值日人员

#### 3.3.2 学期模式

- 在2月28日或8月30日自动启用学期模式
- 学期模式按学期显示周数（春季学期：2/28-7/31，秋季学期：8/30-次年2/18）
- 非学期模式下按整年显示周数

### 3.4 考勤评价功能

#### 3.4.1 评价表单

**功能描述**：
管理员在列表视图下点击成员后弹出评价表单，可以记录该成员当天的值日情况。

**表单字段**：
1. **出勤状态**（必选）
   - 已到（present）
   - 缺席（absent）
   - 不合格（fail）
   - 待改进（improve）
   - 待定（pending）
   - 优秀（perfect）
2. **评分**（可选）
   - 范围：0-4分
   - 0分表示未评分
   - 支持键盘上下箭头调整
   - 默认值（选择状态后自动填入）：
     - 缺席：1分
     - 不合格：1分
     - 待改进：2分
     - 已到：3分
     - 优秀：4分
     - 待定：0分（不计入统计）

3. **惩罚天数**（可选）
   - 支持手动设置
   - 根据状态和评分自动计算
   - 可手动调整

4. **代值功能**（可选）
   - 勾选"代值"表示该成员的值日由他人代替
   - 选择代值人
   - 原值日人员分数为0

5. **还值功能**（可选）
   - 如果该成员之前代替过他人值日，可以标记还值
   - 选择还值对象（必须是之前代替过的人）
   - 还值后减少代值次数

#### 3.4.2 评价规则

**自动计算规则**：
- 缺勤：第一次1天，后续2天；周五额外+1天
- 不合格（评分=1）：基础1天，后续2天

**特殊情况**：
- 全体缺勤：整组成员自动标记为缺勤，并计算惩罚
- 重大活动：标记后该天值日失效，不影响统计
- 补值：额外值日可以减免惩罚天数（1天），但表现不合格仍会追加惩罚

### 3.5 额外值日人员管理

#### 3.5.1 添加额外值日

**功能描述**：
管理员可以给任意日期添加额外值日人员，用于补值或惩罚性值日。

**添加方式**：
- 在列表视图中点击"添加"按钮
- 选择日期和人员以及值日原因（若选择代值，补加对应被代值人员，若还值，选择对应人员，补值则正常）
- 确认添加

**补值减免机制**：
- 如果该人员有惩罚天数，添加额外值日后自动减免1天
- 补值日期的表现如果不合格，仍会追加惩罚天数

#### 3.5.2 删除额外值日

- 管理员可以删除已添加的额外值日人员
- 删除后去除对应的考核评价

### 3.6 低分预警功能

#### 3.6.1 预警显示

**功能描述**：
在列表视图顶部显示低分预警，提醒关注低分成员。

**预警条件**（须同时满足）：
- 平均分大于0且小于2分
- 至少有2条有效评分记录（仅1条记录不触发预警）

**显示内容**（当前实现）：
- 成员姓名、所属组别
- 均分、缺席次数、不合格次数
- （可选扩展：最近一次低分记录的日期；颜色区分：平均分<2分显示红色，小于2.5分显示黄色）

#### 3.6.2 预警更新

- 实时更新，每次评分后重新计算
- 只显示当前存在的低分成员

### 3.7 统计数据查看

#### 3.7.1 统计页面入口

- 管理员可以通过导航栏的"考核统计"按钮进入统计页面
- 普通用户无法访问统计页面

#### 3.7.2 统计数据展示

**统计表格**：
- 组别：显示成员所属组别，并使用组别标签颜色区分不同小组（例如小组1蓝色、小组2绿色等）
- 分组可视化：当表格从一个小组切换到另一个小组时，使用明显的分隔线进行视觉分组

- 平均分：计算所有有效评分（>0）的平均值，无评分显示"-"
- 考勤情况：显示出勤、待改进、不合格、缺席、优秀的次数，通过点击对应成员的按钮后额外通过文本框显示
- 惩罚天数累计：只累加正向惩罚天数，不因补值减免而减少；显示格式如 `+3 天`
- 惩罚天数-正常：所有记录的 `penaltyDays` 代数和（含补值的负值减免），正值红色、负值绿色
- 代值列表：若该成员曾作为代值人代替他人值日，则显示"代值 N 人"按钮，点击后弹出被代替人员名单
- 操作：查看详细记录（仅终端管理员可访问）

**筛选功能**：
- 支持按组别筛选
- 支持查看全部成员

#### 3.7.3 详细记录查看

**功能描述**：
点击"查看记录"按钮，弹出该成员的所有考勤记录详情。

**显示内容**：
- 记录列表：按日期倒序显示
- 每条记录显示：
  - 日期
  - 出勤状态
  - 评分
  - 惩罚天数
  - 备注信息
- 终端管理员可以删除记录，删除记录后更改该属性对应数据，如惩罚天数累计，惩罚天数-正常，平均分，考勤情况

### 3.11 代指和换值功能详解

#### 3.11.1 代指功能

**使用场景**：
当某成员无法值日时，由其他成员代替值日。

**操作流程**：
1. 管理员在评价表单中勾选"代值"
2. 选择代值人（不能选择本人）
3. 保存后：
   - 原值日人员的分数为0，状态不变
   - 记录代值关系

**代指记录**：
- 系统记录每个成员代值过的人员列表以及每个成员被代值的人员列表
- 记录每个成员的代值次数以及被代值的次数

#### 3.11.2 还值功能

**使用场景**：
当某成员之前代替过他人值日，现在由他人代替自己值日来"还值"。

**操作流程**：
1. 管理员在评价表单中勾选"还值"
2. 选择还值对象（必须是之前代替过的人）
3. 保存后：
   - 减少原成员的代值次数
   - 更新代值记录

**限制条件**：
- 只能对之前代替过的人员进行还值
- 还值对象的列表自动筛选

## 4. 用户权限管理

### 4.1 权限等级设计

#### 4.1.1 普通用户权限

普通用户（未登录或未具有管理员身份的用户）主要用于**查看信息**，不具备任何写入或修改能力。

- 允许的操作：
  - 查看首页值日排班（日历视图）
  - 查看每周值日组及组内成员
  - 查看每天每个人的值日考勤情况（只读）
  - 查看统计页面中对外开放的汇总信息（如后期开放只读统计页时）
- 禁止的操作：
  - 不能修改任何考勤记录、评分或惩罚天数
  - 不能调整值日组、不能添加/删除额外值日人员
  - 不能访问管理员管理页、统计管理页等后台页面

#### 4.1.2 普通管理员权限

普通管理员是在终端管理员授权下创建的账号，负责日常的值日考勤记录与评价工作。

- 额外允许的操作：
  - 登录管理员后台（通过“管理员登录”点击后的弹窗）
  - 对每天每个值日成员进行考勤评价（出勤状态、评分、惩罚天数）
  - 设置代指、还值、补值等特殊记录
  - 标记全体缺勤、重大活动日
  - 调整某一周的值日组（排班覆盖）
  - 添加、删除额外值日人员
  - 查看并使用“考核统计”页面（但无权清空所有记录）
  - 修改自己的密码
- 仍然禁止的操作：
  - 不能创建或禁用其他管理员账号
  - 不能查看或撤销管理员操作日志
  - 不能删除他人考勤记录（仅终端管理员可做彻底删除）

#### 4.1.3 终端管理员权限

终端管理员是系统最高权限账户，默认账号为 **ZRWY**，默认密码为 **goodday**。该账号在系统设计中被视为**不可删除、不可禁用**的超级管理员。

- 拥有的权限包括：
  - 拥有普通管理员的所有权限
  - 创建新的管理员账号（设置用户名和密码）
  - 禁用、启用普通管理员账号
  - 查看所有管理员的操作日志
  - 删除某些管理员操作（如错误禁用、误添加等）
  - 查看所有管理员密码（用于线下交接与紧急排查）
- 不允许的操作（规则约束）：
  - 不能删除自身账号
  - 不能禁用自身账号

#### 4.1.4 权限继承关系

系统采用自上而下的权限继承模式：

- 普通用户：仅有“查看”权限
- 普通管理员：**普通用户权限 + 业务操作权限（评价、排班调整等）**
- 终端管理员：**普通管理员权限 + 管理员管理与高危运维权限**

通过这种分层结构，可以保证：

- 日常使用集中在普通管理员层面，降低风险
- 高危操作集中在少数终端管理员手中，便于审计与追责
- 普通用户始终处于只读状态，保证系统数据稳定

### 4.2 登录认证机制

#### 4.2.1 登录流程

1. 普通用户点击首页右上角“管理员登录”按钮，弹出登录弹窗。
2. 在登录弹窗中输入：
   - 账户名称（username）
   - 密码（password）
3. 前端通过 Supabase 进行身份验证：
   - 查询 `admins` 表（或使用 Supabase Auth）校验用户名、密码及禁用状态
   - 仅当账号存在且未被禁用时视为验证通过
4. 验证通过后：
   - 在前端内存状态中记录当前登录用户信息（用户名、是否为终端管理员等）
   - 由 Supabase 维护实际会话信息，界面状态据此显示“考核统计”、“管理员管理”、“修改密码”等入口
5. 验证失败时：
   - 提示“账号或密码错误”或“账户已被禁用”
   - 不写入任何登录状态

#### 4.2.2 密码验证

当前版本密码验证逻辑在前端本地完成：

- 管理员账户数据结构包含：`username`、`password`、`isRoot`、`isDisabled` 等字段
- 登录时按照用户名精确匹配账户，然后比对明文密码
- 若 `isDisabled === true`，即使密码正确也不允许登录
- 终端管理员账号在逻辑中被特殊保护：
  - 初始化时即存在
  - 不允许被删除或标记为禁用

（后续版本可以扩展为后端校验与加密存储，这里策划书主要描述现有前端实现。）

#### 4.2.3 会话管理

- 登录成功后，前端在内存状态中保存当前管理员信息：
  - `username`：当前管理员账号
  - `isRoot`：是否为终端管理员
- 页面初始化（如刷新）时：
  - 默认视为未登录管理员，需要重新输入账号密码
  - 根据是否存在有效会话/内存状态决定是否显示“考核统计”、“管理员管理”、“修改密码”等入口
  - 显示“管理员登录”或“退出管理”按钮
- 退出登录时：
  - 清除前端内存中的管理员信息
  - 重置前端 `isAdmin`、`isRootAdmin` 状态，并根据需要调用 Supabase 注销接口

#### 4.2.4 安全措施

在当前“前端 + Supabase”架构下，安全措施主要体现在：

- **权限前端控制**：所有修改类功能均检查 `isAdmin` / `isRootAdmin` 状态
- **终端管理员保护**：
  - 无法通过界面删除或禁用 ZRWY 账号
  - 无法在管理员列表中对其执行禁用操作
- **操作确认**：高风险操作提供二次确认弹窗，例如：
  - 禁用/启用管理员
  - 清空所有考勤记录
- **日志记录**：重要的管理员管理操作（添加、禁用、启用）都会被写入操作日志，支持事后审计

### 4.3 权限控制实现

#### 4.3.1 前端权限控制

前端通过全局状态与条件渲染来实现界面级的权限控制：

- 关键状态：
  - `isAdmin`：当前是否为管理员登录状态
  - `isRootAdmin`：当前管理员是否为终端管理员
- 页面与组件根据上述状态决定：
  - 是否展示“管理员管理”、“考核统计”、“修改密码”等入口
  - 在成员列表中展示“评价/修改”按钮还是“查看”按钮
  - 是否显示重置数据、清空记录等高危操作入口

#### 4.3.2 操作权限验证

在具体操作函数中再次进行权限校验，防止仅通过隐藏按钮来“伪安全”：

- 评价相关操作：仅在 `isAdmin === true` 时允许执行 `handleAttendanceSave`、`handleGroupAbsent` 等
- 统计页面访问：若未检测到 `adminUser`，会自动重定向回首页
- 清空所有记录：仅当 `isRootAdmin === true` 时展示并允许执行
- 删除单条记录：仅终端管理员在记录详情弹窗中拥有删除按钮

通过“按钮层 + 逻辑层”双重校验，尽量避免误触发和越权操作。

#### 4.3.3 数据访问控制

- 所有写入 Supabase 的关键数据（考勤记录、管理员信息等）都只通过受控的服务函数或统一逻辑修改
- 非管理员用户：
  - 无法通过正常交互触发任何写入操作
  - 只能读取排班信息、考勤展示信息，不会暴露管理员密码等敏感字段
- 管理员相关敏感信息（如密码）仅在管理员管理页面的受控区域内显示，且默认以“••••••”形式隐藏

### 4.4 管理员账户管理

#### 4.4.1 账户创建流程

- 入口：终端管理员登录后，在“管理员管理”页面中使用“添加新管理员”表单
- 流程：
  1. 输入新管理员的账户名称与密码
  2. 点击“添加管理员”按钮
  3. 前端调用管理员服务逻辑：
     - 校验账户名是否重复
     - 按约定结构写入到supabase设置的对应数据列
     - 记录一条“ADD_ADMIN”类型的操作日志
  4. 刷新管理员列表显示，提示“管理员添加成功”

#### 4.4.2 账户禁用机制

- 禁用对象：仅限普通管理员账号，终端管理员账号不可禁用
- 禁用效果：
  - `isDisabled` 字段设置为 `true`
  - 被禁用账号无法再登录，即使密码正确
  - 管理员列表中以“已禁用”标签高亮显示
- 操作流程：
  - 终端管理员在列表中点击“禁用”按钮
  - 弹出确认对话框，确认后执行禁用逻辑
  - 写入“DISABLE_ADMIN”类型的操作日志

#### 4.4.3 操作日志记录

操作日志主要用于追踪管理员管理相关行为，便于审查与回滚：

- 记录内容包括：
  - 日志ID
  - 操作时间戳
  - 操作管理员用户名
  - 操作类型（ADD_ADMIN / DISABLE_ADMIN / ENABLE_ADMIN / REVERT 等）
  - 操作详情（如被操作的管理员账号）
  - 是否已撤销及撤销人信息
- 展示方式：
  - 在“管理员管理”页底部，以表格形式展示
  - 支持终端管理员对尚未撤销的操作执行“撤销”

#### 4.4.4 密码修改功能

- 适用对象：普通管理员（终端管理员一般不在前端修改密码）
- 入口：管理员登录后，点击“修改密码”按钮，弹出密码修改弹窗
- 核心逻辑：
  - 验证旧密码是否正确
  - 校验两次输入的新密码是否一致
  - 更新本地管理员数据中的密码字段
  - 可选：记录一条密码修改相关的日志（仅记录操作用户，不记录新密码）

## 5. 值日排班管理

### 5.1 排班规则设计

#### 5.1.1 默认轮换规则

- 系统预设 8 个小组（小组1~小组8），每组 3 人
- 以周为单位进行排班，**每一周对应一个值日组**
- 默认规则：
  - 从系统设定的起始周开始，按小组1 → 小组8 的顺序循环
  - 当到达小组8后，再从小组1重新开始
- 默认排班在前端根据周序号动态计算，无需后端存储完整排班表

#### 5.1.2 排班覆盖机制

考虑到现实中临时调班、调整周次的需求，系统设计了“覆盖规则”：

- 覆盖数据结构：
  - 使用 `scheduleOverrides` 映射：`weekStartISOString -> groupId`
  - 仅记录“与默认规则不同”的周次
- 覆盖逻辑：
  1. 计算该周默认值日组（按轮换规则）
  2. 检查 `scheduleOverrides` 中是否存在该周的覆盖记录
  3. 若存在覆盖记录，则以覆盖的 groupId 为准
  4. 若覆盖 groupId 无效（组被删除或改名），回退到默认组并记录错误日志
- 存储方式：
  - 覆盖数据保存在 `Supabase.schedule_overrides` 表中（通过 API 读写）
  - 支持随时恢复默认规则（删除 `schedule_overrides` 表中对应记录）

#### 5.1.3 周末处理规则

- 周一~周五：直接使用当前周的值日组
- 周六、周日：
  - 默认沿用**本周的值日组**，即周末值日与工作日同组
  - 在列表视图中，可通过“包含周末”开关决定是否展示周末
- 特殊场景：
  - 若某天被标为重大活动日，则该日的值日统计失效
 
### 5.2 排班调整功能

#### 5.2.1 周排班修改

- 管理入口：
  - 在列表视图中每周标题区域，管理员可看到“本周值日组”下拉选择框
- 操作流程：
  1. 管理员从下拉框中选择新的值日组
  2. 系统将该周的 `weekStart` 日期转换为 ISO 字符串作为键
  3. 写入或更新 `scheduleOverrides[weekKey] = newGroupId`
  4. 将 `scheduleOverrides` 保存到 Supabase 对应的排班覆盖表（如 `schedule_overrides`）
  5. 触发页面重新渲染，显示最新排班结果
- 使用场景：
  - 调整某周为指定小组（如考试周、活动周等）
  - 修正原本轮换顺序中的临时变更

#### 5.2.2 额外值日人员

额外值日人员属于**附加在原有排班上的个体级别调整**，常用于：

- 对表现不佳成员的惩罚性补值
- 对有惩罚天数成员的减免安排

实现方式：

- 额外值日数据结构中记录：`memberId` + `dateISO`
- 在某天的人员列表中：
  - 先根据该日值日组拿到“常规成员列表”
  - 再将额外值日成员合并进当日成员列表，并标记 `isExtra = true`
- 展示：
  - 在列表视图中，对同时是常规成员又是额外成员的情况，增加“(额外添加)”标识
  - 管理员可针对额外成员单独删除该条额外值日

#### 5.2.3 排班历史记录（策划）

当前实现中，排班历史主要通过以下数据间接反映：

- `scheduleOverrides`：记录了哪些周曾被调整过值日组
- 考勤记录：通过某日的成员与状态可推断当时的实际排班

如需进一步增强排班历史的可视化展示，可在后续迭代中增加：

- “排班历史”视图：按时间轴列出每周的实际值日组和变更记录
- 对每次排班调整写入一条“排班调整日志”（包括操作人、时间、前后对比）

### 5.3 排班显示优化

#### 5.3.1 日历视图显示

- 使用月历组件（如 react-big-calendar 或自研月视图）展示整月排班
- 每个日期格中：
  - 显示值日组名称
  - 展示当天所有值日成员以及其考勤状态
  - 对周末值日采用弱化视觉，但逻辑一致
- 管理员在日历视图中可以：
  - 通过点击日期添加额外值日成员
  - 通过点击成员打开评价弹窗

#### 5.3.2 列表视图显示

- 以“周”为最小单位的纵向列表，更适合做精细管理与日常记录
- 优化点：
  - 所有与同一周相关的操作集中在一个卡片内，减少跳转
  - 每天的操作（全体缺勤、重大活动、重置、添加人员）放在日期行最醒目的位置
  - 当天的代指/还值信息以“值日调整信息”模块集中展示，避免分散阅读

#### 5.3.3 日期导航功能

- 在列表视图中使用“年份 + 周数”组合的周选择器：
  - 支持快速切换到任一年份的任一周（覆盖该年全部周次，通常为 52 或 53 周）
  - 每个选项显示该周完整日期范围，例如“第5周（2月2日～2月8日）”
  - **跨年周特殊处理**：若某周跨越两个自然年（如年末首周），则在日期中补全年份，明确显示为“第1周（2025年12月29日～2026年1月4日）”，消除跨年歧义
- “包含周末”开关：
  - 勾选时：一周显示 7 天
  - 关闭时：仅显示周一到周五，减少信息噪音
  - 该开关默认关闭
- 在日历视图中提供：
  - 年份选择器、月份选择器
  - “今天”按钮，一键返回当前日期所在月份

## 6. 考勤评价系统

### 6.1 评价表单设计

#### 6.1.1 表单字段设计

考勤评价表单是管理员日常操作的核心界面，字段设计既要**信息完整**，又要**输入简洁**：

- 基本字段：
  - 成员姓名（只读，显示当前被评价成员）
  - 日期（只读，显示当前评价日期）
- 状态与评分：
  - 出勤状态：下拉选择【已到 / 缺席 / 不合格 / 待定 / 优秀 /】
  - 评分：0–4 分的数字输入框，支持键盘上下键调整
  - 惩罚天数：非负整数输入框，可置 0 或自动计算值
- 特殊标记：
  - 代值（复选框 + 代值人选择）
  - 还值（复选框 + 还值对象选择）
  - （可选扩展）备注文本框：记录特殊说明

#### 6.1.2 数据验证规则

为了保证考勤数据的有效性，表单在提交时需要进行以下校验：

- 出勤状态：必填，必须为预定义枚举之一
- 评分：
  - 允许为空（表示暂未评分，记为 `null`）
  - 若填写，必须为 0–4 的整数
  - 对于缺勤状态，系统自动将评分视为 1 分
- 惩罚天数：
  - 必须为整数，且不小于 0
  - 若为空则按 0 处理
- 代值逻辑：
  - 勾选“代值”时，必须选择代值人，且代值人不能是当人
  - 提交后自动为代指人生成对应的惩罚记录
- 还值逻辑：
  - 仅当当前成员存在代指历史记录时，才可以启用“还值”
  - 勾选“还值”时，必须从“曾经代替过的人”列表中选择对象

#### 6.1.3 用户体验优化

为减少填写负担、避免误操作，表单在交互层面进行以下优化：

- 根据出勤状态自动给出**合理默认分数**：
  - 出勤：默认 3 分
  - 不合格：默认 1 分
  - 缺席：默认 1 分
  - 待改进： 默认 2 分
  - 优秀： 默认 4 分
- 当评分在 1 分时，界面提示“可能触发惩罚天数，请确认”
- 评分输入框支持键盘上下键微调，并限制在 0–4 范围内
- 提交前弹出摘要确认（可选）：例如“张三 / 3月5日 / 出勤 / 1分 / 惩罚1天”
- 对代值、还值操作在界面上用醒目的提示文字说明其含义，避免误用

### 6.2 评价记录管理

#### 6.2.1 记录创建流程

1. 管理员在列表或日历视图中点击某个成员
2. 系统弹出考勤评价弹窗，若当日已有记录则自动回填
3. 管理员填写/修改状态、评分、惩罚天数及代指/还值等信息
4. 提交后：
   - 若该成员当日已有记录：更新原记录（保持原记录 ID）
   - 若无记录：创建新记录，生成唯一 ID
5. 所有记录统一写入 Supabase.attendance_records表（通过 API 读写）

#### 6.2.2 记录修改机制

- 修改入口与创建入口共用同一弹窗
- 逻辑上以“成员ID + 日期”作为主键查找记录：
  - 若存在则加载为当前状态
  - 管理员修改后重新保存
- 更新规则：
  - 覆盖写入新的状态、评分、惩罚天数等字段
  - 若未显式修改的字段（如是否重大活动、全体缺勤标记），按业务规则决定是否保留或重置

#### 6.2.3 记录删除权限

- 普通管理员：
  - 不具备**彻底删除**记录的能力
  - 若需“重置某天”，通过业务逻辑将该天相关成员记录清空，但不提供单条删除接口
- 终端管理员：
  - 在“考核统计”页面的“记录详情”弹窗中，可以删除单条记录
  - 删除后立即更新本地存储与统计数据
  - 可选：为删除操作记录一条日志，以备追踪

#### 6.2.4 记录查询功能

当前版本主要通过“统计页面 + 详情弹窗”来实现记录查询：

- 在统计页面中：
  - 先按组别或全部维度筛选成员
  - 再点击某成员的“查看记录”按钮
- 在记录详情弹窗中：
  - 以时间倒序展示该成员的所有考勤记录
  - 支持终端管理员删除某一条记录

后续可扩展：

- 增加按日期范围、状态、惩罚天数等维度的多条件过滤
- 提供导出 CSV/Excel 的功能，便于线下归档（仅终端管理员）

### 6.3 特殊状态处理

#### 6.3.1 全体缺勤标记

- 使用场景：某天该组所有值日成员均未参加值日
- 操作方式：
  - 管理员在某一天的日期行点击“缺”按钮
  - 若当前为“非全体缺勤”状态，则：
    - 为该日所有值日成员生成或更新记录：状态为缺席、评分0分、按规则计算惩罚天数，并标记 `isGroupAbsent = true`
  - 若再次点击（已是全体缺勤）：
    - 将所有成员当日状态重置为“待定”，惩罚天数归 0，`isGroupAbsent = false`

#### 6.3.2 重大活动标记

- 使用场景：如大型考试、学校活动等导致当日值日制度暂时失效
- 操作方式：
  - 管理员点击某日的“活”按钮
  - 若当前不是重大活动日：
    - 为所有相关成员记录 `isImportantEvent = true`
    - 该日期视为“值日无效日”，不参与惩罚与成绩计算
  - 若再次点击：
    - 取消 `isImportantEvent` 标记，恢复为正常待定状态（无评分）

#### 6.3.3 待定状态处理

- “待定”状态用于：
  - 未来日期尚未发生的排班
  - 当天尚未完成评价的成员
  - 被重大活动或全体缺勤标记后，恢复时的中间状态
- 处理规则：
  - 待定记录不计入评分平均值
  - 待定记录不产生惩罚天数
  - 在界面上以灰色标识，提醒管理员“尚未完成评价”

## 7. 惩罚与评分机制

### 7.1 惩罚规则设计

#### 7.1.1 缺勤惩罚规则

- 当成员状态为“缺席”（absent）时：
  - 第一次缺勤：基础惩罚天数为 1 天
  - 第 N 次缺勤（N ≥ 2）：基础惩罚天数为 2 天
  - 若缺勤日为周五或被规则视为“周五等价日”，则额外 +1 天（详见 7.1.2）
- 缺勤记录默认评分为 1 分，且不需要手动填写


#### 7.1.2 特殊日期惩罚（周五）

为强调周五值日的重要性，系统对周五缺勤设置了更严格的惩罚：

- 当缺勤日是周五时：
  - 在已有基础惩罚的基础上额外 +1 天
- 若周五当天被标记为重大活动日：
  - 惩罚规则允许将“周五缺勤惩罚”向前转移到同一周内最接近的非重大活动工作日（如周四、周三）
  - 转移逻辑通过专门的“特殊日惩罚计算函数”实现

#### 7.1.3 代指自动惩罚

对于代值行为，系统采用“放任自由”的设计：

- 代值人：
  - 其当日评分按实际表现计算
  - 记录被代值人欠对方一次值日
- 被代值人：
  - 记为 0 分或待定，根据实际情况由管理员判断
  - 是否产生惩罚根据当日状态设定

### 7.2 自动计算算法

#### 7.2.1 惩罚天数计算函数

系统将惩罚计算逻辑封装为独立函数，便于维护与复用：

- 输入参数通常包括：
  - 考勤状态（absent / fail）
  - 日期对象（用于判断星期几与特殊日）
  - 缺勤次数 / 不合格次数
  - 评分（score）
  - 当前所有考勤记录（用于统计历史）
- 输出结果：
  - 本次记录应增加的惩罚天数（可为正数、0）

#### 7.2.2 特殊日期检测

在惩罚计算中，特殊日期（如周五或由重大活动造成的转移日）需要单独处理：

- 通过 `date.getDay()` 获取当前是周几（0–6）
- 在周五或其前几日中，结合重大活动标记判断“惩罚实际生效日”
- 若当日为特殊日，则在基础惩罚的基础上 +1 天

#### 7.2.3 累计次数统计

为了区分“第一次缺勤”和“多次缺勤”，需要统计成员历史记录：

- 在计算缺勤惩罚前：
  - 过滤出该成员所有历史记录
  - 统计状态为缺席的记录数，即为 `absentCount`
  - 将 `absentCount` 作为参数传入惩罚函数
- 对不合格次数、低分次数等也可以采用类似统计方法，以支持更复杂的惩罚策略（如累计三次不合格触发额外惩罚）

### 7.3 手动调整机制

#### 7.3.1 手动调整界面

在考勤评价表单中，惩罚天数字段始终允许手动编辑：

- 管理员可以在系统自动计算的基础上：
  - 将惩罚天数调低（例如酌情减轻）
  - 将惩罚天数调高（例如严重违规）
  - 设置为 0（例如特殊说明下免罚）
- 表单中应通过说明文字提示：
  - “留空或填 0 将按系统规则自动计算”
  - “手动填写将覆盖系统计算结果，包括惩罚结果累计中涵盖的那一份数据”

#### 7.3.2 调整记录保存

- 手动调整后的惩罚天数会直接写入当前考勤记录的 `penaltyDays` 字段
- 之后的统计逻辑只会读取该字段，不再重新计算
- 若需要审计，可在后续版本中：
  - 为手动调整增加“修改原因”字段
  - 或将原始自动计算结果存入单独字段以备对比

#### 7.3.3 调整权限控制

- 只有管理员（含普通管理员与终端管理员）可以修改惩罚天数
- 普通用户在查看弹窗中只能看到结果，不能编辑
- 若未来引入“多级审批”，可以限制：
  - 普通管理员仅能在自动结果基础上做±1天的微调
  - 终端管理员可以进行大范围调整

### 7.4 补值减免机制

#### 7.4.1 补值识别逻辑

系统通过“额外值日人员”记录来识别补值行为：

- 若某成员在非本组轮换日被额外安排值日，则认为存在“补值”可能
- 在评价时：
  - 检查该日是否存在该成员的额外值日记录
  - 若存在，则将本次记录标记为“补值记录”

#### 7.4.2 自动减免规则

当成员因惩罚而进行补值时，系统自动给予适当减免：

- 在添加额外值日时：
  - 统计该成员历史惩罚总天数
  - 若总惩罚天数 > 0，则自动创建一条“补值减免记录”：
    - `penaltyDays = -1`（减 1 天）
    - 备注提示“补值自动减免一天惩罚”
  - 将该减免记录也写入考勤记录中，参与惩罚天数-正常的计算，但不计入惩罚天数累积的计算

#### 7.4.3 补值不合格处理

为了防止“形式上的补值”，系统对补值表现不佳的情况进行额外约束：

- 若补值记录的状态为“不合格”或评分等于 1 分：
  - 在原有处罚基础上追加惩罚天数（例如 2 天）

通过这一机制，促使被惩罚成员在补值时真正认真履行值日职责，而不是“走过场”。

## 8. 统计与数据分析

### 8.1 统计数据计算

#### 8.1.1 平均分计算

- 统计维度：以“成员”为统计单位，计算其所有有效评分的平均分。
- 有效评分定义：
  - `score !== null` 且 `score > 0` 的记录
  - 0 分通常代表“未评分”或占位，不计入平均值
- 计算逻辑：
  1. 根据成员 ID 过滤出其全部考勤记录
  2. 过滤出有效评分记录
  3. 使用「平均分 = 有效评分总和 ÷ 有效记录数」公式计算
  4. 若有效记录数为 0，则平均分显示为 `"-"`
- 展示：
  - 在统计页面中，以 1 位小数显示，例如 `1.3`、`3.0`
  - 通过颜色区分不同水平：
    - ≥ 3.0 分：绿色
    - 2.0-2.9 分：橙色
    - 1.0-2.0 分：红色

#### 8.1.2 惩罚天数累积

- 统计对象：所有记录中 `penaltyDays` 字段存在且不为 `undefined` 的记录
- 计算逻辑：
  - 对该成员所有记录的 `penaltyDays` 进行求和
  - 支持正值：
    - 正值：新增惩罚天数
  - 汇总后的 `totalPenaltyDays` 代表该成员**当前总惩罚天数**（理论上应补值的总次数，不进行减少）
- 展示：
  - 在统计页中以 `+3天`、`+0天` 等形式显示

#### 8.1.3 惩罚天数-正常
- 统计对象：所有记录中 `penaltyDays` 字段存在且不为 `undefined` 的记录
- 计算逻辑：
  - 对该成员所有记录的 `penaltyDays` 进行求和
  - 支持正值：
    - 正值：新增惩罚天数
  - 汇总后的 `totalPenaltyDays` 代表该成员**当前总惩罚天数**（理论上应补值的总次数）
- 展示：
  - 在统计页中以 `+3天`、`+0天` 等形式显示


#### 8.1.3 考勤次数统计

- 对每个成员，统计其各类状态出现的次数：
  - 已到/优秀（present）：正常完成值日的次数
  - 待改进（improve）：值日但是表现一般
  - 不合格（fail）：表现不佳的次数
  - 缺席（absent）：未到岗的次数
- 计算方法：
  1. 按成员 ID 过滤出考勤记录
  2. 使用计数器分别统计四种状态的数量
  3. 将结果存入 `attendanceCount` 字段：`{ present, improve, absent, fail }`
- 展示：
  - 以带有颜色圆点的小标签形式展示在“考勤情况”一列
  - 方便快速对比同组内不同成员的出勤表现

### 8.2 数据筛选功能

#### 8.2.1 按组别筛选

- 在统计页面提供组别下拉框：
  - `全部`：显示所有组别成员
  - `小组1` ~ `小组8`：仅显示对应小组成员
- 实现逻辑：
  - 先计算出所有成员的统计数据
  - 再根据所选组别对结果做一次过滤

#### 8.2.2 时间范围筛选（可选扩展）

当前实现中统计默认基于“全时段”数据，后续可扩展为：

- 指定起止日期，仅统计该时间范围内的记录
- 典型应用：
  - 按学期评优（例如春季学期、秋季学期）
  - 按月份或自定义区间查看表现
- 实现方式：在计算前对原始记录按日期先行过滤

#### 8.2.3 状态筛选（可选扩展）

- 可在界面上增加筛选条件，如：
  - 只看“平均分低于 2 分”的成员
  - 只看“惩罚天数大于 0”的成员
  - 只看“不合格次数 ≥ N”的成员
- 目的：快速锁定需要重点关注或约谈的对象

### 8.3 数据可视化

#### 8.3.1 表格展示

- 主统计页面以表格形式呈现所有成员的关键指标：
  - 组别、姓名、平均分、考勤次数、惩罚天数、操作
- 表格特点：
  - 支持滚动，适应较多成员
  - 表头固定，便于对照
  - 行高适中，信息密度与可读性平衡

#### 8.3.2 颜色标识

- 使用统一的颜色语义系统：
  - 绿色：优秀 / 正向表现（高分、已到）
  - 橙色：中等 / 需要留意（及格边缘、不合格次数）
  - 红色：警告 / 需要重点关注（低分、缺席、较多惩罚天数）
- 目的：
  - 在不阅读具体数字的情况下，也能一眼看出哪一列、哪一行存在问题

#### 8.3.3 排序功能（可选扩展）

- 可考虑在后续版本中支持按以下字段排序：
  - 平均分（从高到低 / 从低到高）
  - 惩罚天数（从多到少）
  - 缺席次数（从多到少）
- 这样可以在评优或整改时快速筛出“Top N”或“Bottom N”成员

### 8.4 详细记录查看

#### 8.4.1 记录列表展示

- 在统计表格中点击“查看记录”按钮后，弹出成员详情弹窗：
  - 以时间倒序展示该成员所有考勤记录
  - 每条记录至少包含：日期、状态、评分、惩罚天数
- 便于：
  - 追溯某次惩罚来源
  - 查阅某段时间连续表现情况

#### 8.4.2 记录详情查看

- 对单条记录，可展示更多信息（如果数据中存在）：
  - 是否为代指/还值记录
  - 是否为补值减免记录
  - 是否处于重大活动日
  - 备注信息（如存在）
- 如有需要，可以扩展为点击单条记录进入“只读版评价弹窗”，完整查看当时的录入信息

#### 8.4.3 记录删除功能

- 删除权限：仅限终端管理员，可在详情弹窗中看到“删除”按钮
- 删除逻辑：
  - 从 `attendanceRecords` 中移除对应 ID 的记录
  - 立即刷新统计数据与界面
  - 可选：记录删除行为到管理员操作日志中
- 使用场景：
  - 误操作录入错误记录
  - 修正严重偏离实际情况的数据

## 9. 技术架构设计

### 9.1 前端技术栈

#### 9.1.1 React 18 框架

- 使用 React 18 作为基础前端框架，提供组件化 UI 构建能力

#### 9.1.3 TypeScript 类型系统

- 全面使用 TypeScript 对核心数据结构（成员、组别、考勤记录、管理员等）进行类型约束

  - 提高开发时的自文档性
  - 提前发现类型错误，减少运行时异常
  - 为后续系统扩展提供良好的基础

#### 9.1.4 TailwindCSS 样式框架

- 使用 TailwindCSS 进行原子化样式编写：
  - 快速构建页面布局与组件样式
  - 减少独立 CSS 文件数量，统一视觉风格
- 统一主题色与状态色，例如：
  - 主题蓝：`#2a63b7`
  - 警告红：`#ff2300`
  - 成功绿：`#00bd39`

### 9.2 组件架构

#### 9.2.1 页面组件

  - `admin-management.tsx`：管理员管理后台
- 页面组件负责：路由层逻辑组合、整体布局与权限入口控制

#### 9.2.2 功能组件

- 放在 `components` 目录下，按功能拆分：
  - 日历与排班相关：`DutyCalendar`、`CalendarView`
  - 考勤评价相关：`AttendanceModal`、低分提示组件等
  - 管理相关：`LoginModal`、`ChangePasswordModal`、`AdminManagement`

- 功能组件聚焦具体业务模块，避免页面组件过于臃肿

#### 9.2.3 通用组件与复用

- 通用弹窗、按钮、选择器等组件尽量复用，减少重复代码
- 通过 props 控制组件行为，实现：
  - 读写模式切换（如 `AttendanceModal` 的 `readOnly` 参数）
  - 管理员/普通用户不同展示逻辑

#### 9.2.4 组件间数据流

- 数据主要由顶层页面组件管理，再以 props 向下传递：
  - 例如首页维护 `attendanceRecords`、`extraDutyMembers` 等状态
  - 将相关数据与回调函数传给日历与列表组件
- 子组件调用回调更新上层状态，上层统一写入 supabase，实现单一数据源

### 9.3 状态管理

#### 9.3.1 React Hooks 使用

- 使用 `useState` 管理视图状态与业务状态：当前日期、视图模式、登录状态等
- 使用 `useEffect` 处理：
  - 初始化时从 Supabase API 读取数据
  - 状态变化后同步写回 Supabase API
  - 特殊时间点（如 2/1、8/1）触发学期模式逻辑

#### 9.3.2 本地状态管理

- 不引入 Redux 等复杂全局状态库，而是依托组件树结构与 props 传递即可满足需求
- 对于跨组件复用的数据与逻辑（如值日状态 Hook），抽离成自定义 Hook 提高复用性

#### 9.3.3 数据持久化

- 通过 Supabase 数据库 持久化关键数据：
  - 考勤记录
  - 额外值日人员
  - 值日排班覆盖
  - 管理员账户
  - 当前登录管理员信息
- 页面加载时优先从 supabase 数据库读取，若无则使用默认数据或空数组初始化

### 9.4 工具库使用

#### 9.4.1 date-fns 日期处理

- 用于：
  - 周起始日期与结束日期计算
  - 月份、年份切换
  - 判断日期是否在区间内
  - 生成学期范围内所有周
  - 格式化日期展示（如 `yyyy年MM月dd日`、`M月d日`）

#### 9.4.2 react-big-calendar 日历组件（如使用）

- 提供月视图、周视图等基础能力
- 通过自定义事件渲染，将每天的值日组与成员状态渲染到日历格子中
- 使用 `eventPropGetter` 自定义事件的背景色、边框等样式

#### 9.4.3 其他依赖库

- TailwindCSS：实现整站布局与样式风格
- Next.js 内置路由与构建工具：简化开发和部署流程

## 10. 数据存储方案

### 10.1 存储方式选择

#### 10.1.1 Supabase 作为后端数据库

- 选型理由：
  - 提供托管 PostgreSQL 数据库与完整的 CRUD API
  - 内置身份认证与行级安全（RLS），方便做权限控制
  - 前端可通过官方 JS SDK 直接访问，无需自建后端
- 优点：
  - 多设备、多浏览器之间数据实时共享
  - 数据存储在云端，不受本地缓存清理影响
  - 可水平扩展，支持更大数据量与更多用户
- 可接受的成本：
  - 需要配置 Supabase 项目与安全策略
  - 需要在部署环境中配置环境变量（URL、anon key 等）

#### 10.1.2 前端与 Supabase 的交互方式

- 使用官方 `@supabase/supabase-js` SDK：
  - 在前端初始化单例客户端：`createClient(SUPABASE_URL, SUPABASE_ANON_KEY)`
  - 通过 `from('<table>').select/insert/update/delete` 进行数据操作
- 读写策略：
  - 页面初始化时批量读取需要的核心数据（考勤记录、排班覆盖、管理员信息等）
  - 用户操作（新增/修改/删除）时，先更新前端状态，再调用 Supabase API 持久化
  - 关键操作失败时给予错误提示，并视情况回滚前端状态

#### 10.1.3 简单缓存与降级策略

- 主数据源为 Supabase，localStorage 仅作为**可选的只读缓存**：
  - 可缓存最近一次从 Supabase 拉取的只读快照（例如统计结果），用于极端网络不佳时展示
  - 任何写操作必须成功同步到 Supabase 才视为成功，不再依赖 localStorage 作为真实数据源
- 网络异常时的显示策略：
  - 连接 Supabase 失败时，统一在界面显眼位置提示“云端数据暂不可用”
  - 若存在本地缓存，可提供“仅查看最近缓存数据”的只读模式

### 10.2 数据结构设计（Supabase 表）

#### 10.2.1 考勤记录表 `attendance_records`

- 主要字段：
  - `id`：主键（UUID）
  - `date`：日期（date / timestamp）
  - `member_id`：成员 ID（string）
  - `status`：考勤状态（enum：present/absent/fail/improve/pending/perfect 等）
  - `score`：评分（int，0–4）
  - `penalty_days`：惩罚天数（int，可为负数表示减免）
  - `is_substituted`、`substituted_by`、`is_exchanged`、`exchanged_with`
  - `is_important_event`、`is_group_absent`、`is_compensation`
  - `created_at`、`updated_at`：时间戳
- 索引与优化：
  - 按 `member_id + date` 建立联合唯一索引，保证“同一成员同一日期仅一条记录”
  - 按 `date` 建立索引，提升按日期范围查询性能

#### 10.2.2 管理员表 `admins`

- 主要字段：
  - `id`：主键（UUID）
  - `username`：账号名（唯一）
  - `password`：密码（当前可先明文存储，正式环境需改为哈希）
  - `is_root`：是否为终端管理员
  - `is_disabled`：是否被禁用
  - `created_at`、`disabled_at`、`disabled_by`、`enabled_at`、`enabled_by`
- 登录状态：
  - 当前会话信息不再长久保存在 localStorage 中，而是优先使用 Supabase Auth 或短期会话缓存
  - 如需简单实现，可在前端维护内存状态 + 浏览器 SessionStorage（会随浏览器关闭清除）

#### 10.2.3 排班覆盖表 `schedule_overrides`

- 主要字段：
  - `id`：主键（UUID）
  - `week_start`：周起始日期（ISO 字符串或 date）
  - `group_id`：覆盖后使用的小组 ID
  - `created_by`：操作管理员 ID
  - `created_at`：创建时间
- 说明：
  - 仅记录与默认轮换不同的周次，保持表数据量较小

#### 10.2.4 额外值日表 `extra_duties`

- 主要字段：
  - `id`：主键（UUID）
  - `member_id`：额外值日成员 ID
  - `date`：额外值日日期
  - `reason`：原因（惩罚补值 / 主动补值 / 其他）
  - `is_compensation_used`：是否已用于减免惩罚
  - `created_by`、`created_at`

### 10.3 数据操作流程

#### 10.3.1 数据读取

- 页面初始化时：
  - 调用 Supabase API 读取当前学期/时间范围内的考勤记录、排班覆盖、额外值日与管理员列表
  - 读取结果写入 React 状态，作为前端单一数据源
- 查询注意事项：
  - 尽量按日期范围与成员分组分页查询，避免一次性拉取全部历史数据
  - 对统计页可单独触发按需查询，减少首页负载

#### 10.3.2 数据写入与更新

- 新增 / 修改考勤记录：
  - 前端先在内存中更新对应记录
  - 调用 `supabase.from('attendance_records').upsert(...)` 持久化
- 新增 / 修改管理员：
  - 仅终端管理员可操作，对应调用 `admins` 表的 insert/update
- 排班覆盖与额外值日：
  - 分别写入 `schedule_overrides` 与 `extra_duties` 表
- 失败处理：
  - Supabase 操作返回错误时，需提示用户，并回滚前端状态或重新拉取数据

#### 10.3.3 数据删除

- 删除单条考勤记录：
  - 仅终端管理员可通过统计详情页操作
  - 调用 `supabase.from('attendance_records').delete().eq('id', ...)`
- 清空某日或某周记录：
  - 通过按日期范围批量删除相应考勤与额外值日记录
- 删除排班覆盖或额外值日：
  - 对应删除 `schedule_overrides`、`extra_duties` 中的记录

### 10.4 数据备份与迁移

#### 10.4.1 Supabase 层备份

- 优先使用 Supabase 提供的数据库备份与导出能力：
  - 定期自动快照（由 Supabase 项目配置决定）
  - 手动导出某段时间内的主要业务表数据（CSV/SQL）

#### 10.4.2 应用层导出

- 在管理员后台提供“导出当前统计数据”入口：
  - 按当前筛选条件导出成员统计为 CSV / Excel
  - 可选导出单个成员的详细考勤记录

#### 10.4.3 迁移与环境切换

- 若后期需要从测试环境迁移到正式环境：
  - 通过导出测试库数据，在正式库中导入
  - 在部署环境中切换 Supabase URL / Key 环境变量
- 若从旧版 localStorage 方案迁移到 Supabase：
  - 提供一次性迁移脚本，将浏览器中的旧数据结构整理后写入 Supabase
  - 迁移完成后不再依赖 localStorage 作为数据源

## 11. 界面设计与用户体验

### 11.1 设计原则

#### 11.1.1 简洁性原则

- 界面以“任务为中心”，用户一眼就能看到本周值日安排和操作入口
- 避免在同一屏幕叠加过多高级选项，高级功能通过折叠/弹窗呈现

#### 11.1.2 一致性原则

- 所有页面使用统一的字体、字号、颜色与按钮样式
- 操作区布局尽量统一（例如“保存/取消”按钮始终在弹窗右下角）
- 数据密集页面（如“考核统计”、“周列表视图”、“月历视图”）整体采用 `text-sm` 及以上字号；评价弹窗内容字号为 `text-sm`，标题为 `text-base`，确保可读性
- 已评价成员在周列表视图中以绿色背景高亮并显示 ✓ 标记，方便快速识别遗漏评价的成员

#### 11.1.4 数据可读性原则

- 在统计列表中，优先保证“分组边界清晰”和“核心字段可快速扫读”
- 对组别信息同时使用“文本 + 颜色 + 分割线”三种视觉线索，降低误读概率

#### 11.1.3 易用性原则

- 尽量减少完成一项常见任务所需的点击次数：
  - 如一周内所有成员评价都能在单个页面完成
- 对关键按钮提供鼠标悬浮提示（title），解释其作用（如“全体缺勤”、“重大活动”等）

### 11.2 颜色方案

#### 11.2.1 主题色设计

- 主题色：蓝色 `#2a63b7`，用于标题、主按钮、重要链接
- 辅助背景：浅灰 `#f5f5f5`，搭配白色卡片，营造整洁感

#### 11.2.2 状态颜色映射

- 状态与颜色：
  - 出勤良好：绿色
  - 待改进： 蓝色
  - 不合格：黄色
  - 缺席：红色
  - 待定：灰色
 
- 分数等级和惩罚情况使用同一套颜色语义，避免混淆

#### 11.2.3 对比度要求

- 保证文字与背景对比充足，可在大屏、投影等环境下清晰可见
- 禁用按钮、提示文本等使用浅灰，但仍保持可辨认性

### 11.3 交互设计

#### 11.3.1 操作流程优化

- 针对管理员的常见流程（如“选周 → 选日 → 选人 → 评价”）设计顺手的路径
- 重要入口（统计、管理员管理）集中放在顶部导航区域

#### 11.3.2 反馈机制

- 对危险操作（禁用管理员、清空记录等）增加二次确认弹窗
- 保存成功/失败通过明显提示（alert 或 toast）反馈

#### 11.3.3 错误处理

- 表单错误（如未填写必要字段、分值超范围）在对应字段下方给出红色错误提示
- 对于不可预期错误（如解析失败），友好提示“数据异常，可尝试刷新或联系维护人员”

### 11.4 响应式设计

#### 11.4.1 移动端适配

- 列表视图在窄屏下采用单列布局，卡片纵向排布
- 控件（按钮、选择框）触控区域足够大，避免点不中

#### 11.4.2 平板适配

- 可同时显示周列表和当天详情，减少来回切换

#### 11.4.3 桌面端优化

- 为桌面端设计完整的多栏布局：
  - 左侧为周/月份导航
  - 中间为主内容（列表或日历）
  - 右侧可用于显示提示、说明或统计摘要

## 12. 部署与维护

### 12.1 部署方案

#### 12.1.1 Vercel 部署（推荐）

- 适用于 Next.js 项目的一键部署方案：
  - 将代码推送到 GitHub
  - 在 Vercel 导入项目，选择 Next.js 预设
  - 使用默认构建命令 `npm run build`，输出 `.next`
  - 部署完成后获取在线访问地址

#### 12.1.2 自有服务器部署

- 一般流程：
  1. 在服务器上安装 Node.js（版本 ≥ 18）
  2. 拉取代码并安装依赖 `npm install`
  3. 构建项目 `npm run build`
  4. 使用 `npm start` 或 PM2 等方式常驻运行
  5. 使用 Nginx 等进行反向代理和域名绑定

#### 12.1.3 Docker 容器部署

- 通过 Dockerfile 构建镜像：
  - 基于 Node 18 Alpine 镜像
  - 拷贝代码并安装依赖
  - 构建并暴露 3000 端口
  - 使用 `docker run -p 3000:3000` 启动

#### 12.1.4 静态导出部署（可选）

- 若未来选择将项目改造为纯静态站点（不依赖 SSR 功能）：
  - 配置 `next.config.js` 使用 `output: 'export'`
  - 通过 `next export` 生成 `out` 目录
  - 将静态文件部署到任意静态托管服务（如 OSS、COS、GitHub Pages 等）

### 12.2 环境配置

#### 12.2.1 Node.js 版本要求

- 建议使用 Node.js 18.17.0 及以上版本，以满足 Next.js 14 的最低要求

#### 12.2.2 依赖安装

- 使用 npm：`npm install`
- 关键依赖包括：Next.js、React、TypeScript、TailwindCSS、date-fns 等

#### 12.2.3 构建配置

- 构建命令：`npm run build`
- 开发环境启动：`npm run dev`
- 生产环境启动：`npm start`

### 12.3 维护指南

#### 12.3.1 数据备份建议

- 由于业务数据统一存储在 Supabase（PostgreSQL）中，部署侧需要关注云端数据库的备份策略
- 建议：
  - 启用或定期检查 Supabase 项目的自动备份/快照配置
  - 定期由终端管理员在后台导出关键业务数据（如统计报表、原始记录）为 CSV/Excel 进行线下备份

#### 12.3.2 版本更新流程

- 本地修改代码并通过 Git 做版本管理：
  - 使用分支进行开发与测试
  - 合并到主分支后再触发部署（Vercel 或自有 CI 流程）
- 更新前建议：
  - 在测试环境验证数据库表结构与数据兼容性（如 Supabase 表字段新增、重命名、删除等）
  - 涉及表结构变更时，优先通过数据库迁移脚本或 Supabase 控制台进行灰度迁移，而不是直接在生产上修改

#### 12.3.3 故障排查

- 常见问题：
  - 页面无法加载：检查构建是否成功、服务器或 Vercel 日志
  - 数据异常：检查与 Supabase 的网络连接及 API 调用是否报错，可在浏览器控制台查看请求与响应
  - 权限异常：检查 Supabase 的行级安全（RLS）策略与 `admins` 表中对应管理员状态是否正确；若使用 Supabase Auth，还需检查登录会话是否已过期

### 12.4 安全建议

#### 12.4.1 密码安全

- 当前版本可在 Supabase 的 `admins` 表中以明文或简单加密形式存储密码，仅适用于内部或教学场景
- 若用于正式环境，推荐：
  - 使用 Supabase Auth 或自建后端进行密码哈希存储与校验，不在数据库中保存明文密码
  - 前端不直接保存明文密码，仅持有短期会话令牌（如 Supabase 提供的 access token）

#### 12.4.2 数据安全

- 避免在公共设备上长期保持管理员登录状态，使用完毕及时退出
- 对生产环境的 Supabase 项目：
  - 妥善保管 `SUPABASE_URL` 和 `SUPABASE_ANON_KEY` 等环境变量，不在代码仓库中提交服务端密钥
  - 区分开发/测试/生产环境的 Supabase 实例，避免误操作生产数据

#### 12.4.3 访问控制

- 建议将终端管理员账户仅限少数人掌握，并线下妥善保管
- 禁止随意分享部署地址给无关人员，尤其是带有真实成员姓名与记录的环境
